# 内容归属规则

这篇用于防止知识库越写越杂。遇到新资料时先判断“它能回答什么问题”，再决定放置位置。

## 一句话规则

| 内容回答的问题 | 放置位置 |
|---|---|
| 这个业务正常怎么走，失败从哪些分叉切？ | `20_Service-Flows` |
| 当前现象第一轮应该先查什么？ | `30_Troubleshooting` |
| 这个问题真实发生过，第一坏点和证据是什么？ | `40_Case-Library` |
| 代码入口、状态同步、产物链路在哪里？ | `50_Platform-Code` |
| APN/ECC/NV/CarrierConfig/运营商名怎么配、怎么验证生效？ | `60_Configuration` |
| log 关键字、抓取工具、解析工具怎么用？ | `70_Tools-Debug` |
| 术语、cause code、缩写和基础概念是什么？ | `10_Basics` |

## 容易混的边界

| 边界 | 放置判断 |
|---|---|
| Registration vs Data | Attach / TAU / PLMN / 默认承载建立前放 Registration；Attach 成功后上网、DNS、TCP、MMS、业务 PDN 放 Data |
| Registration vs IMS | LTE/NR 注册、默认承载和 AP ServiceState 放 Registration；IMS APN、P-CSCF、SIP REGISTER、VoLTE 能力放 IMS |
| Data vs Configuration | 真实失败链路放 Data case；APN 字段、匹配规则、数据库和生效验证放 Configuration |
| Call vs IMS | CS call、ECC、SRVCC、EPSFB、补充业务入口放 Call；SIP 注册、VoLTE/VoWiFi/VoNR 媒体和 IMS 能力放 IMS |
| Configuration vs Platform-Code | “怎么配、怎么 dump 生效值”放 Configuration；“哪个类/模块读取配置、状态如何同步”放 Platform-Code |
| Tools-Debug vs Case | 可复用 log 字段和工具方法放 Tools-Debug；某次真实问题的关键证据放 Case |

## Case 状态值

| status | 含义 | 使用建议 |
|---|---|---|
| `raw_import` | 仅导入，还未精修 | 不作为可靠结论引用 |
| `partial` | 有可用片段，但证据链不完整 | 可沉淀配置/字段，不宜写强 root cause |
| `summarized` | 已压缩成可复用 case | 默认推荐状态 |
| `summarized_from_search` | 基于查询摘要整理，未拿到完整详情 | 结论写边界，后续可补详情 |
| `summarized_with_log_gap` | 有结论但缺少关键 log 或对比证据 | 排障时可参考，不可当最终定论 |
| `closed` | 历史问题已关闭且结论可用 | 可引用，但仍保留证据边界 |
| `open` | 问题仍在跟踪或待补证据 | 不写成最终 root cause |
| `reference` | 资料型 case 或规则引用 | 用于导航，不当真实失败样例 |
| `evidence_requirement` | 证据要求或判定标准 | 用于约束结论口径 |

新增 case 优先使用 `summarized`、`partial`、`raw_import`、`open`。旧状态保留兼容，但不要继续扩散无定义状态。

Case Library 里允许保留 `doc_type: reference` 或 `doc_type: index` 的集合入口，但它们不算真实 case。`Case横向索引` 只统计 `doc_type: case`，集合入口要在 README 或精修清单中说明去向。

## 文档质量等级

所有 Markdown 文件都必须有文件顶部 frontmatter，至少包含 `doc_type`、`domain`、`status`、`quality`。

| quality | 含义 | 要求 |
|---|---|---|
| `curated` | 已整理成可直接引用的主文档 | 顶部有明确入口、结论或定位原则；正文按流程/配置/排障组织 |
| `imported_reference` | 从语雀、旧知识库、CQ 或附件迁入的参考资料 | 必须有“阅读入口/使用方法/速查结论/拆分说明”，说明该怎么读、哪些内容只作溯源 |
| `generated` | 脚本生成文件 | 不能手工维护，必须写清生成脚本 |
| `template` | 模板文件 | 只放写作结构，不放真实问题结论 |

导入资料允许保留原始截图、外链和长步骤，但必须在顶部说明推荐阅读方式。可复用结论要继续沉淀到主流程、配置方法、排障流程或独立 case，不能让“补充资料”无限变长。

## CQ / 外部资料导入规则

1. 不新建“CQ 导入”大目录，必须落到现有业务域、配置、工具或平台代码结构里。
2. CQ 详情中联系人、电话、邮箱、紧急联系人不进入知识库。
3. 只有“真实发生 + 有第一坏点 + 有关键证据”的内容才建 case。
4. 只有字段口径、配置方法或排障经验的内容，直接沉淀到 `60_Configuration`、`30_Troubleshooting` 或 `70_Tools-Debug`。
5. 没有明确结论的内容可以写成“待确认/证据要求”，不能写成事实。

## HTML 兼容规则

Markdown 给我看，HTML 给人看，所以写法要同时兼容两边：

- 表格内不要使用带显示名的 Obsidian wiki 链接，改用 `[label](target.md#anchor)`。
- 表格单元格里如果必须出现竖线，用 `/` 或文字说明替代。
- 图片路径必须是本地附件或可稳定访问的链接。
- 长 log 不直接贴入表格，放代码块或 case 证据段。
