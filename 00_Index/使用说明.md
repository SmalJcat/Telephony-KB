# 使用说明

## 每次遇到问题怎么沉淀

1. 先从 [[../30_Troubleshooting/常见问题速查流程|常见问题速查流程]] 判断现象类别。
2. 到 [[业务知识地图]] 找对应业务域入口。
3. 用 [[内容归属规则]] 判断这份材料应该进 case、流程、配置、代码还是工具。
4. 如果是真实问题，复制 [[../99_Templates/Case模板|Case模板]]，填项目、版本、SIM、运营商、RAT、场景。
5. 用 [[../99_Templates/时间线模板|时间线模板]] 拉出 AP log、modem log、bugreport 的统一时间线。
6. 标出第一个异常点，只保留能证明结论的关键证据。
7. 把结论写成“事实 / 推断 / 待确认”三类。
8. case 结束后，把可复用经验补回流程、配置、代码架构、排障或工具文档。
9. 新增或修改 case 后重新生成 [[../40_Case-Library/Case横向索引|Case横向索引]]。

## 推荐命名

Case 文件建议使用：

```text
YYYY-MM-DD_领域_现象_根因.md
```

示例：

```text
2026-05-12_IMS_注册失败_SIP403.md
2026-05-12_Registration_LTE无服务_TA-not-allowed.md
2026-05-12_SIM_开机不识卡_UICC状态异常.md
```

## 文档分层

目录分层统一维护在 [[../README#目录分层|README]]。这里不重复目录说明，只记录内容放置判断。

## 快速判断放哪里

| 你手里的内容 | 放置位置 |
|---|---|
| “这个问题真实发生过，log 里第一坏点是...” | `40_Case-Library` |
| “这个流程正常应该先 A 后 B，失败从哪几个分叉看” | `20_Service-Flows` |
| “遇到无服务、数据失败、掉话时先查什么” | `30_Troubleshooting` |
| “这个状态由哪个模块产生，代码从哪里看” | `50_Platform-Code` |
| “这个运营商配置、APN、ECC、NV 怎么配” | `60_Configuration` |
| “这个 log 关键字、字段、抓取方式是什么” | `70_Tools-Debug` |
| “这是术语、cause、缩写、标签” | `10_Basics` |

详细边界见 [[内容归属规则]]。

## 写作要求

- 结论短，证据硬。
- 不确定就写“推断”或“待确认”。
- 不把大段 log 贴进文档，只保留关键行和字段解释。
- Case 的 frontmatter 必须从文件第一行开始，字段参考 [[../10_Basics/CauseCode与缩写#标签体系|标签体系]] 和 [[../99_Templates/Case模板|Case模板]]。
- 每个 case 至少要有：现象、时间线、异常点、证据、结论、下一步。
- 每次复盘都问一句：下次遇到同类问题，我应该先查什么？
- 大幅新增或迁移文档后，运行 [[../70_Tools-Debug/知识库维护工具#Case横向索引|Case横向索引生成]]、[[../70_Tools-Debug/知识库维护工具#HTML导出|HTML导出]] 和 [[../70_Tools-Debug/知识库维护工具#知识库健康检查|知识库健康检查]]。
