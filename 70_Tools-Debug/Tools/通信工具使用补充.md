 ![](../../attachments/outline/c61abdf0-c926-4ea3-a9fd-9f723406f654.png)

路径下会生成以下文件，分析音频文件

 ![](../../attachments/outline/126487ce-f9ba-4588-adcc-d58bf8630532.png)


分析思路如下，

UL0-->UL1-->UL

UL0：mic录制上来的数据--->有问题需要mobile log和寄存器or tone音实验确认。

UL1：mic上来经过算法处理数据。算法为MTK 算法or (hifi3)三方通话算法  --->有问题需要tuning算法参数or 算法团队确认。

UL：发送给网络的数据  --->有问题先确认是否按了静音按钮，若没有mute需要modem team 确认。


DL-->DL0-->DL1

DL：网络下来的数据  --->有问题需要modem team 确认。

DL0：网络下来经过解码后数据  --->有问题需MTK 协助确认。

DL1：解码后经过算法处理数据。算法为MTK 算法or (hifi3)三方通话算法  --->有问题需要tuning算法参数or 算法团队确认。

### NVTool 读机失败先核对 AP/CP 版本匹配

CQWeb `SPCSS01010613` 记录过一个容易误判的工具问题：升级 4G modem 版本后，NVTool 无法从手机 `Load from Phone` 读取 NV。历史分析结论是 AP 基线与 4G modem 版本不匹配，不能直接按 NVTool、端口或 NV 分区损坏处理；合入匹配 AP patch 后读机恢复正常。

排查顺序：

1. 先记录 AP 基线、modem base version、modem build time 和 NVTool 版本。
2. 确认当前 modem 是否来自该 AP 基线配套版本，临时 modem 包要特别标注。
3. 如果 NVTool 连接或读机失败，同时 modem boot 状态异常，先做 AP/CP 版本匹配确认。
4. 只有版本匹配后仍失败，才继续看端口、DIAG、dump、fixnv 分区和工具版本。

### fastboot 刷 fixnv 操作边界

CQWeb `SPCSS00766912` / `SPCSS00967511` 记录过同类问题：使用 fastboot 直接刷 `w_fixnv1_a` 或 `l_fixnv1_a` 后，IMEI 和校准参数丢失。

典型高风险命令：

```text
fastboot flash w_fixnv1_a xxx_base_nv_nvitem.bin
fastboot flash l_fixnv1_a xxx_customer_nvitem.bin
```

操作约束：

- 原生 fastboot 不一定支持 fixnv 的 NV backup/merge，不能把公共 `nvitem.bin` 当成现场机备份。
- 量产、售后或客户现场机如果必须刷 fixnv，先确认 bootloader/fastboot 已集成 `backupnv` 或 NV merge patch。
- 刷前保留 `fixnv1/fixnv2` 主备分区镜像；刷后验证 IMEI、RF 校准、SIM 识别、紧急呼叫等依赖个体化 NV 的功能。
- 更推荐使用 OTA 或下载工具完成带参数保留能力的升级/刷机流程。

### A/B OTA 中断点与 nvmerge

CQWeb `SPCSS01234705` 补充了 OTA/FOTA 里的关键判断：A/B OTA 对 `fixnv` 的处理要看 `update_engine` 日志，而不是只看是否已经提示重启。

关键日志：

```text
update_engine: Applying 1 operations to partition "l_fixnv1"
update_engine: Applying 1 operations to partition "l_fixnv2"
update_engine: Running "/postinstall/bin/nvmerge 1 3"
```

操作约束：

- `l_fixnv1/l_fixnv2` 写入后、`nvmerge` 开始前，是个体化 NV 还未完成合并的风险窗口。
- 如果在这个窗口掉电、强制关机或接入 ResearchDownload / DeviceKit 线刷，需要保留 `update_engine` 日志和 fixnv 主备回读证据。
- 如果已经走到提示重启，通常不能再按“nvmerge 前中断”复现，需要重新设计 A->B->A 或 B->A->B 的中断测试。
