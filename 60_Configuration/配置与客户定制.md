## CarrierConfig与客户定制

CarrierConfig 和客户定制通常决定“功能是否允许出现、是否允许注册、是否允许走某条策略路径”。它不总是根因，但经常是问题触发条件。

定位时按这个顺序：

1. 确认 SIM 的 MCC/MNC、GID、SPN、IMSI 是否符合预期。
2. 查看 `dumpsys carrier_config` 中当前生效配置。
3. 对比项目默认配置、运营商覆盖配置、客户定制配置。
4. 确认配置变化是否发生在问题前后。
5. 对能力类问题，确认 framework、IMS service、modem 三层能力是否一致。

字段级 key 和默认值不要在本页维护，统一看 [CarrierConfig参数映射](CarrierConfig参数映射.md)。

## APN / SPN配置速查

APN 问题转 [APN配置方法](APN配置方法_重构.md)，运营商名称和 SPN/PNN/OPL 问题转 [运营商名称配置方法](运营商名称配置方法.md)。

最容易混淆的边界：

| 现象                              | 应看                                               |
| ------------------------------- | ------------------------------------------------ |
| default / MMS / IMS / XCAP 建链失败 | APN 类型、profile、PDN reject、IA APN 下发              |
| 状态栏运营商名错误                       | SPN/PNN/OPL/NITZ/operator XML、`showSpn/showPlmn` |
| 漫游图标不符合预期                       | CarrierConfig 的 roaming / non-roaming 配置         |
| 手动搜网列表名称错误                      | PLMN 列表来源、PNN/OPL、平台列表显示逻辑                       |

## ECC配置速查

ECC 配置、号码来源和运行时验证统一看 [ECC配置方法_重构](ECC配置方法_重构.md)。

排查时先区分：

| 场景 | 关键证据 |
|---|---|
| 无卡紧急呼叫 | 本地 emergency database、radio/modem 可用 RAT、实际 phoneId/slotId |
| 有卡紧急呼叫 | EF_ECC、网络下发、EmergencyNumberTracker、拨号前 AP 判定 |
| 双卡号码共享 | 卡槽、phoneId、subId、号码来源和 AP 下发目标 |
| PIN/PUK 锁卡 | EF_ECC 是否可读、域选择、modem call domain 判断 |
