# 小区广播配置方法

## 速查结论

- 小区广播（Cell Broadcast / CBS / CB）配置要先区分 Android Mainline/full 版本、Go/非 Go、RTOS MMI、modem/L4 上报和 UI 展示层。
- Android 13 之后部分小区广播能力已经 Mainline 化，默认总开关、标题、音频、振动时长等不一定能通过项目源码直接客制化。
- 信道测试必须使用目标地区 MCC/MNC 的白卡、实卡或仪表配置；用 `00101` 白卡不能证明目标国家/运营商信道配置生效。
- LTE 紧急广播类信道可能不受普通小区广播菜单开关完全控制；如果产品要求不展示，需要明确是在接收层关闭还是 MMI/UI 层过滤。

## 模板化定位

### 配置来源

| 来源 | 本文落点 | 运行时验证 |
|---|---|---|
| CellBroadcast 配置 | channel range、MCC/MNC、开关默认值 | CB app / cellbroadcast log |
| Mainline / full 版本 | 版本限制、资源位置差异 | APK/模块版本、配置是否进入目标包 |
| modem / network | LTE/NR CB/CMAS/ETWS 上报 | modem CB trace、RIL indication |

### 匹配与生效链路

```text
配置 channel / MCCMNC / 开关
-> CellBroadcast 模块加载
-> modem 上报 CB / CMAS / ETWS
-> AP 过滤和展示
```

### 平台差异

| 平台 | 重点看点 | 验证口径 |
|---|---|---|
| Android common | AOSP 公共 XML、Provider、framework 读取点 | 先证明 common 默认值和运行时 dump 是否一致 |
| UNISOC | carrier overlay、CarrierService、Operator NV、modem profile | 同时看 AP log、产物配置、NV/readback 和 modem trace |
| MTK | vendor/mediatek 私有配置、SBP/DSBP/CXP、NVRAM | 结合 debuglogger、ELT/MD log、AP dump 验证最终值 |
| Qualcomm | CarrierConfig overlay、MCFG/QCRIL、modem profile | 结合 dumpsys、QXDM/QCAT、MCFG 产物确认 |

### 验证命令与 log

| 目标 | 证据入口 | 预期结论 |
|---|---|---|
| 源配置存在 | CellBroadcast 配置 / channel range / emergency alert rules | 能定位到需求字段、默认值和项目覆盖值 |
| 运行时 dump 生效 | CellBroadcastService、CBS settings、logcat alert flow | 设备当前值与预期配置一致 |
| AP/vendor 已采用 | Telephony/RILJ/vendor service log | 能看到读取、选择、下发或业务判断动作 |
| modem/协议侧采用 | CB message 上报、ETWS/CMAS trace | 协议字段、modem 状态或 reject cause 能与配置结果闭环 |

## 配置边界

| 场景 | 优先判断 | 典型结论 |
|---|---|---|
| 默认总开关 | Android 版本、Mainline 状态 | Android 13 上历史结论是总开关默认值不可修改 |
| 信道列表 | full / Go 版本、MCC/MNC、channel range | full 版本闭源时不能按源码修改信道 |
| 标题/铃声/振动时长 | Mainline 资源、overlay 是否可覆盖 | Android 16 历史结论是运营商要求需向 Google 提 issue |
| 关闭菜单仍收告警 | 信道类型、RAT、是否紧急广播 | LTE 下 4370 紧急广播可能仍上报，MMI 可选择过滤展示 |
| 收不到广播 | 底层是否上报、L4/MMI 是否收到 | 先搜底层/协议栈上报关键字，再看 UI |

## Android Mainline / full 版本限制

从 CQWeb 历史问题 `SPCSS01099451`、`SPCSS01192688`、`SPCSS01606659` 看，小区广播配置不能只按旧平台改 `packages/apps/CellBroadcastReceiver/res/values/config.xml`。

关键限制：

- Android 13 上小区广播 Go 和非 Go 已经 Mainline 化，总开关默认值历史结论是不可修改。
- full 版本 `com.google.android.cellbroadcastreceiver` 闭源时，无法通过修改源码来客制化信道。
- Android 16 上标题、音频和振动时长如果受 Mainline 控制，本地 overlay 或反编译修改不一定生效；运营商强制要求时应向 Google 提 issue 或走正式合规方案。

验证建议：

1. 先确认设备使用的是 AOSP/UNISOC 源码版、Google full 版还是 Mainline 模块。
2. 再确认修改点是项目源码、overlay、Google 模块资源还是 modem/RTOS MMI。
3. 若修改 `config.xml` 不生效，不要继续盲改 Java；先判断该模块是否已闭源或 Mainline。
4. 量产需求不要使用临时 sample code，需要正式 patch 或正式版本确认。

## 信道与 MCC/MNC 测试

小区广播信道配置通常和地区、运营商、RAT、告警类型有关。历史问题中，使用 `00101` 白卡测试目标地区信道时出现大量 NOK；结论是需要使用目标地区 MCC/MNC 的白卡、实卡，或把仪表配置成目标 MCC/MNC。

排查要点：

| 检查项 | 说明 |
|---|---|
| SIM / 白卡 MCC/MNC | 必须匹配目标法规或运营商要求 |
| 仪表 MCC/MNC | 不只写 SIM，仪表广播侧也要匹配目标地区 |
| values-mcc 资源 | 删除所有地区资源后统一改全局配置属于高风险改法，需完整回归 |
| channel range | 确认新增信道和默认信道是否都被 set 到 modem |
| log 证据 | 同时看配置加载、channel enable、底层上报、UI 展示 |

## 关键日志

Android / AP 侧常见关键字：

```text
CellBroadcastReceiver
CellBroadcastConfigService
setCellBroadcastRange
additional_cbs_channels_strings
com.google.android.cellbroadcastreceiver
```

RTOS / L4 / MMI 侧常见关键字：

```text
APP_MN_SMSCB_MACRO_MSG_IND
APP_MN_ETWS_PRIM_IND
```

判断边界：

- 如果底层没有 `APP_MN_SMSCB_MACRO_MSG_IND`，先怀疑广播未收到或未上报，不要只查 UI。
- 如果底层已上报但 UI 不显示，再看 MMI 过滤、菜单开关、语言/编码、弹窗逻辑。
- 如果关闭菜单后仍收到 LTE 紧急广播，先确认是否为强制类告警，再决定是否允许 MMI 层过滤。
