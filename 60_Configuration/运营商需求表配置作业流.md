# 运营商需求表配置作业流

## 速查结论

- 拿到运营商需求表后，先学习和对齐术语，不直接开始配置。
- 默认不维护需求表解析脚本；优先由 AI 人工读取 Excel/PDF 内容、保留原始需求表述，并把每条判断写成可追溯证据。
- 当前默认作业范围只处理 `CarrierConfig` 和 `Modem NV / Operator NV`；APN、ECC、SMS、voicemail、运营商名、小区广播、SIMLock 等先放入 Deferred，除非本次任务明确扩展范围。
- 所有建议必须是默认值差异项：优先使用映射表中的默认值缓存；缓存缺失、冲突、目标分支明显不同或准备落地前，再回目标分支 `CarrierConfigManager.java` / `default.nv` 复核。
- 输出默认只给改动建议和补丁片段，不直接修改平台代码、CarrierConfig XML、modem NV 或 KB，除非用户明确要求写入/落地。

## 输入清单

| 输入 | 必填 | 用途 |
|---|---|---|
| 运营商需求表路径 | 是 | 提取需求项、页签、字段名和原始值 |
| 目标平台 | 是 | MTK / UNISOC / Qualcomm，不同平台默认值和 NV 口径不同 |
| MCC/MNC | 是 | 定位 carrier/operator 匹配条件，注意两位/三位 MNC |
| CarrierConfig 代码路径 | CarrierConfig 项需要 | 校验 key 是否存在并读取 `sDefaults` |
| Modem NV / Operator NV 路径 | NV 项需要 | 校验目标 operator 文件、`default.nv` 和同包默认值 |
| MVNO 条件 | 可选 | GID1、SPN、IMSI prefix、carrier ID 等匹配条件 |
| 期望范围扩展 | 可选 | 明确是否把 APN/ECC/SMS 等纳入本次配置 |

## 人工解析规则

- 不为了单次运营商需求表新增或维护解析脚本；除非用户明确要求自动化，否则不要把工作重心放到脚本、模板或批处理工具上。
- 可以使用表格工具读取 sheet 名、单元格文本和基础结构，但最终分类和取值判断必须由 AI 人工解释，输出中保留需求原文、sheet/字段来源和判断依据。
- 对看似熟悉但口径容易混淆的字段，先查 `References/NV/UNISOC-Modem-NV需求映射.md` 的“人工补充：运营商需求表高频 Needs study 项”，再决定是否配置。
- 不把脚本输出当作结论。脚本或工具最多提供原始文本、行列位置、候选字符串；是否属于 CarrierConfig、Modem NV、Needs study 或 Deferred 必须重新按作业流判断。
- 当需求表字段太多时，先输出“有效改动项 + 默认/已满足/Deferred/Needs study 统计”，再展开明细，避免把大量默认项误认为需要配置。

## 默认值缓存规则

- 日常解析运营商需求表时，先查字段级映射表里的 `Default` / 默认值列，把它作为默认值缓存基线，避免每个字段都回源码搜索。
- CarrierConfig 默认值缓存来自 `60_Configuration/References/CarrierConfig/*.md`；缺少子 bundle 或新 key 时，再查目标 `CarrierConfigManager.java` 并补回映射表。
- UNISOC Operator NV 默认值缓存来自 `References/NV/UNISOC-Default-NV字段映射.md` 和 `UNISOC-Modem-NV需求映射.md`；缺少路径或默认值时，再查目标 modem 包 `OperatorNV/Operator_Cfg/default.nv` 并补回映射表。
- Qualcomm/MTK NV 默认值如果映射表已有，可以用于初筛；但涉及私有 NV、版本差异、工具导出值或 MBN/NVRAM 生成值时，仍要把源码/工具/运行值作为最终证据。
- 必须回源码复核的场景：映射表无默认值、需求值刚好等于/接近默认值但含义不确定、字段在目标分支疑似不存在、目标平台/Android 大版本变化、准备给出可落地补丁、用户明确要求严格按目标分支验证。
- 输出时在 `Default source` 中标明来源：`KB default cache`、`CarrierConfigManager.java`、`default.nv` 或 `Needs study`。使用缓存但未复核源码时，不要写成“已由目标源码确认”。

## 作业步骤

1. 读取需求表结构：列出 sheet、表头、carrier、MCC/MNC、主要功能分组。
2. 建立学习摘要：解释需求表术语、目标平台加载机制、已有证据、候选参数和停止条件。
3. 分类每一行需求：
   - `CarrierConfig`：Android framework/carrier app 可通过 `CarrierConfigManager` 控制的行为。
   - `Modem NV`：modem profile、Operator NV、NVTool、fixnv/RDNV 或协议策略。
   - `Needs study`：看起来属于范围内，但还没有足够证据证明字段或取值。
   - `Deferred`：APN、ECC、SMS、voicemail、运营商名、小区广播、SIMLock 等本次未扩展范围。
4. 先查知识库，再按需查平台代码。知识库用于候选映射、默认值缓存和经验边界；平台代码/default 文件用于补齐缓存缺口或做落地前复核。
5. 建立默认值基线：
   - CarrierConfig：优先使用映射表默认值；缓存缺失或落地前复核时，再确认 key 存在于目标分支 `CarrierConfigManager.java` 并对比 `sDefaults`。
   - UNISOC Operator NV：优先使用 `UNISOC-Default-NV字段映射.md` 默认值；缓存缺失或落地前复核时，再确认非 `PLMN=` 行存在于同一 modem 包 `default.nv` 并对比默认值。
   - Qualcomm/MTK NV：字段映射和默认值缓存只作为初筛，仍需目标分支源文件、产物和运行值确认最终生效。
6. 只保留有效差异项：需求值等于默认值的不新增；已有 carrier/operator 文件已满足的不重复新增。
7. 输出 proposal-only 改动：给出目标文件、旧值、新值、默认来源、补丁片段和风险说明。
8. 给出验证方案：源文件、编译/生成产物、设备运行值、dump/log 证据都要覆盖。
9. 仅在有可复用结论时更新知识库：字段含义、匹配规则、验证方法写入配置文档或 References；真实问题链路写入 Case；工具命令写入 Tools-Debug。
10. 如果更新了映射规则或高频 Needs study 结论，用一份已经人工验收通过的历史需求表做回归抽查，确认输出没有新增误配、默认值重复配置或范围外扩散。

## 输出格式

### 学习摘要

| Topic | Evidence read | Current understanding | Gaps / stop conditions |
|---|---|---|---|
| Requirement vocabulary | 需求表页签/字段 | 术语含义和业务意图 | 模糊表述、冲突值 |
| Platform mechanism | 平台代码、KB | CarrierConfig/NV 加载和匹配方式 | 未确认的 override 顺序 |
| Existing evidence | 现有 XML/NV、default、映射表 | 已满足项和候选字段 | 字段缺失或值编码不明 |
| Default baseline | `sDefaults` / `default.nv` | 哪些值需要配置 | 无法取得目标分支默认值 |
| Runtime verification | dumpsys、NVTool、log | 如何证明最终生效 | 缺少设备/产物/日志入口 |

### 范围摘要

```text
Input:
Target platform:
Carrier / MCC-MNC:
In scope:
Needs study:
Deferred:
Assumptions:
```

### 改动建议

| Req | Category | Parameter | Default source | Default | Existing | Proposed | Confidence |
|---|---|---|---|---|---|---|---|
| 需求原文或编号 | CarrierConfig / Modem NV | key 或 NV path | 文件和行号/来源 | 默认值 | 现有值 | 建议值 | High/Medium/Low |

### 补丁片段

CarrierConfig 用 XML 片段或 diff；NV 用 `old -> new` 和新增行。补丁片段必须能让用户直接定位到目标文件，但默认不替用户写入平台文件。

### 暂不配置项

把跳过原因写清楚：

- 已等于默认值。
- 现有 carrier/operator 文件已满足。
- Key 不存在于 CarrierConfig 映射表或目标 `CarrierConfigManager.java`。
- NV path 不存在于 NV 默认值缓存或目标 `default.nv`。
- 本次范围 Deferred。
- 需要继续研究，不能根据英文需求猜字段。

### 验证方案

CarrierConfig 至少覆盖：

- 源 XML 路径。
- key 存在于 CarrierConfig 映射表或目标 `CarrierConfigManager.java`。
- 建议值不同于映射表默认值缓存或目标 `sDefaults`。
- 编译/安装后的 carrier config 产物。
- `adb shell dumpsys carrier_config --phone <id>` 或匹配日志。

Modem NV 至少覆盖：

- 源 `.nv` / 工具项路径。
- 每条非 `PLMN=` 项存在于 NV 默认值缓存或目标 `default.nv`。
- 建议值不同于映射表默认值缓存或同包默认值。
- 生成后的 NV/bin/image 产物。
- 刷机或打包路径。
- 设备侧运行 NV 读取值或 modem log 证据。

## 质量门禁

| 门禁 | 通过条件 |
|---|---|
| 不直接配置 | 用户只说“配置需求表”时，只输出建议，不改平台文件 |
| 不越界 | 未明确扩展时，APN/ECC/SMS 等放 Deferred |
| 不猜字段 | 每个 CarrierConfig key 和 NV path 都有目标分支证据 |
| 不配默认值 | 需求值等于默认值时不新增 |
| 不重复配置 | 现有 carrier/operator 文件已满足时不重复新增 |
| 可验收 | 每条建议都有默认来源、目标文件、验证方法和风险边界 |
| 不脚本化发散 | 用户未要求自动化时，不新增解析脚本或把脚本结果当最终判断 |
| 可回归 | 映射规则变化后，可用已验收需求表抽查结果稳定性 |
| 默认值可缓存 | 映射表已有默认值时先用缓存；缺失、冲突或落地前再回源码复核，并把新默认值补回映射表 |
