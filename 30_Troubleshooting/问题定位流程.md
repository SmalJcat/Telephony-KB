# 问题定位流程


## 标准六步

1. 定义现象：用户看到什么，不要先写猜测。
2. 定义范围：SIM、注册、IMS、Call、Data、Signal、Stability。
3. 定义时间窗：正常点、触发点、失败点、恢复点。
4. 拉时间线：AP log、radio log、modem log、bugreport统一排序。
5. 找第一个坏点：第一个偏离正常流程的事件。
6. 下结论：事实、推断、待确认分开写。

## 分层判断

| 层级 | 典型证据 | 常见结论 |
|---|---|---|
| AP Framework | ServiceState、IMS callback、RILJ | 状态未更新、策略未触发、配置门控 |
| Vendor RIL/IMS Service | Radio HAL、vendor IMS log | 请求未转发、回调异常、能力不同步 |
| Modem | NAS/RRC/SIP/内部状态 | 信令失败、状态机异常、assert |
| Network | Reject cause、SIP response、Q.850 | 签约、PLMN、TA、IMS核心网 |
| SIM | UICC、EF、IMSI、ISIM | 卡状态、鉴权、应用文件 |

## 最小证据包

不同问题建议至少收集：

| 问题 | 最小证据 |
|---|---|
| 网络注册失败 | AP radio log + modem NAS/RRC log + SIM/PLMN信息 |
| IMS注册失败 | AP IMS/radio log + modem SIP/IMS log + CarrierConfig |
| VoLTE拨号失败 | AP call/IMS log + SIP log + disconnect cause |
| VoWiFi失败 | AP IMS/WFC log + ePDG/IKE log + Wi-Fi网络环境 |
| SIM不识别 | AP UICC log + modem SIM log + slot状态 |
| Modem assert | AP radio log + modem assert dump + 触发前业务场景 |

## 结论强度

- `high`：AP和modem证据一致，且能定位到明确cause或模块。
- `medium`：主链路证据完整，但缺少网络侧或modem内部细节。
- `low`：只有单侧log，只能给出方向。
