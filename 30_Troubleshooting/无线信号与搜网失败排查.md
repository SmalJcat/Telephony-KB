# 无线信号与搜网失败排查

## 速查结论

- “无服务 / 空三角 / 搜不到网 / 数据慢”不要直接归 LTE 注册。先判断是 AP 状态、NAS reject、RRC 建链、RACH、RF 接收、频段能力还是现场弱场。
- LTE reject 后回落 3G/2G 失败时，要把 LTE reject cause 和回落 RAT 的第一坏点分开写。
- 所有 RAT 都失败、手动搜网也失败、对比机同地正常时，优先查 RF 接收链路、天线、RFFE/PA/RFIC 在位和校准。
- 弱场移动类问题必须对齐 AP、modem ARM、DSP/L1 时间点；没有 DSP/L1 证据时不要直接定代码根因。

## 第一轮分层

| 现象 | 关键证据 | 优先方向 |
|---|---|---|
| LTE 找到小区后 Attach Reject | `ATTACH_REJECT`、EMM cause | SIM/签约/TA/PLMN/能力矩阵 |
| LTE reject 后 3G/2G 也失败 | `LOCATION_UPDATING_REQUEST`、`RRCCONNECTIONREQUEST`、`GMMAS_ESTABLISH_REJ` | 回落 RAT 建链或无线侧 |
| 所有 RAT 搜网失败 | `PLMN_SEL_FAILURE_IND`、RSSI/RSRP 掉底 | RF/天线/校准/RFFE |
| 弱场移动后回不来 | RSRP/SNR 很差、频繁 reselection、DSP/ARM 时间不齐 | 覆盖/干扰/回网策略，先补 log |
| 弱场 4G 保持不住 | `MEASUREMENTREPORT` 后 `RRCConnectionRelease redirectedCarrierInfo=geran` | 接收指标低于网络门限，先查 RF/天线/校准和现场弱场差异 |
| 数据已连但很慢 | data call 成功、DNS timeout、TCP retransmission、dbm 低 | 弱网/网络质量，不先改 APN |
| 单一运营商无信号 | 运营商主力 band、项目 band capability | 市场频段/出货配置 |

## 关键日志

### LTE 小区搜索和 NAS reject

```text
MSG_ID_LTE_CPHY_FREQ_SEARCH_CELL_REQ
MSG_ID_LTE_CPHY_FREQ_SEARCH_CELL_CNF
MSG_ID_LTE_CPHY_SYNC_REQ / CNF
MSG_ID_LTEAS_CELL_SELECT_CNF
ATTACH_REQUEST
ATTACH_REJECT emm_cause = NO_SUITABLE_CELL_IN_TA
```

`NO_SUITABLE_CELL_IN_TA` 只说明 LTE 当前 TA 不适合继续服务。后续是否能回 3G/2G，要继续看回落 RAT 建链，不要把所有失败都写成 LTE reject。

### 3G RACH / 建链失败

```text
WRRC_MM_PLMN_CAMPING_CNF
RRCCONNECTIONREQUEST
WRRC_MM_SIGNALLING_ESTABLISH_REQ
WRRC_MM_SIGNALLING_ESTABLISH_CNF est_success = 0
GMMAS_ESTABLISH_REJ
TX_RACH_Preamble
Pre Power=...
nv_max_power:26dbm
net_max_power:24 dbm
```

如果 `RRCCONNECTIONREQUEST` 多次发出但收不到 setup，且 WPHY 显示 RACH preamble 功率已经打到最大，重点查 PA/RF 硬件、校准或单机状态。

### RF 接收掉底

```text
MSG_ID_CMD_ASM_SELECT_CELL_REQUEST
MSG_ID_PLM_AS_3G_PLMN_SEL_REQ
MSG_ID_PLM_AS_GPRS_PLMN_SEL_REQ
MSG_ID_RR_PLM_PLMN_SEL_FAILURE_IND
[MRM] store PCell power level RSRP: earfcn[...] pci[...] rsrp[...]
RSSI main/div path bottom
```

主辅接收长期掉底时，比 NAS/RIL 状态更靠前。此时应优先确认天线扣线、天线弹片、RFFE/PA/RFIC 在位、RF 校准和单机硬件。

## 弱场移动

从 CQWeb 历史问题 `SPCSS01497276` 看，弱场移动后无法回 4G 的问题经常卡在“证据不足”：DUT 目标频点测到约 `-133 dBm`，对比机也只有约 `-120 dBm` 且 SNR 为负，同时 ARM 和 DSP 时间点不完全对齐。这种场景不能只凭 AP 状态或 NAS 事件下结论。

从 CQWeb 历史问题 `SPCSS01502273` 看，弱场下“无法长时间保持 4G”也可能不是 APN/PDN 问题。DUT 在同一 LTE 小区上的 RSRP 明显低于对比机，低于网络配置的异系统测量门限后，网络下发 `RRCConnectionRelease` 并携带 `redirectedCarrierInfo=geran`，UE 被引导到 GSM。遇到这类现象，要把“默认承载是否建立”和“LTE 是否已因射频测量被网络重定向”分开判断。

复测要求：

1. 同卡、同地点、同路线、同时间段对比 DUT 和 REF。
2. 同时保留 AP log、modem ARM log、DSP/L1 log。
3. 记录弱信号区、无信号区、强信号区的进入/离开时间。
4. 记录当前驻留 RAT、EARFCN/UARFCN、PCI/PSC、RSRP/RSRQ/SINR 或 RSCP/ECN0。
5. 如果对比机也是弱场边缘，不要直接判 DUT 平台问题。

## RACH / PA / 校准确认

从 CQWeb 历史问题 `SPCSS01629027` 看，LTE 因 `NO_SUITABLE_CELL_IN_TA` 失败后，DUT 回落到 3G，但 3G 下 RACH preamble 始终发不成功；对比机同卡可完成 3G CS 注册。历史处理方向是检查单机 3G PA 在位、RF 校准和硬件状态。

建议证据：

```text
AT+SGMR=0,0,3,0       // GSM 校准信息
AT+SGMR=0,0,3,1,1     // WCDMA 校准信息
AT+SGMR=0,0,3,3,1     // LTE 校准信息
AT+SPCALICRCCHECK     // 校准数据 CRC 检查
```

如需进一步判断前端器件，可用 Pandora 或平台射频工具读取 RFFE/PA/RFIC 在位状态。

## 频段能力

从 CQWeb 历史问题 `SPCSS01345057` 看，Spectranet 无信号最终归因于出货地区不支持目标运营商所需 Band 40。遇到单运营商无信号时，先确认：

| 检查项 | 判断 |
|---|---|
| 运营商主力 Band | 该运营商在目标区域是否依赖某个项目未开的 Band |
| 项目出货配置 | RF 硬件、modem capability、NV / band mask 是否支持 |
| 对比机 | 对比机是否同样支持目标 Band，不能只看同卡同地 |
| reject cause | `NO_SUITABLE_CELL_IN_TA` 需要结合 band support 判断 |
