# 案例精修清单

这篇只管理 case 的精修优先级，不替代 [Case横向索引](../40_Case-Library/Case横向索引.md)。索引用来查全量 case；本文用来决定下一轮先改哪批。

## 使用入口

当前 [Case横向索引](../40_Case-Library/Case横向索引.md) 只统计 `doc_type: case` 的真实 case；集合入口、外部片段汇总和专题索引用 `doc_type: reference` / `doc_type: index` 管理，不再混入 case 状态统计。

| 分层 | 数量 | 处理策略 |
|---|---:|---|
| `case_summary`：强证据 case | 68 | 有现象、第一坏点、关键证据和结论，作为 HTML 搜索主结果 |
| `supplemental`：补充参考 | 35 | 多为旧库导入或证据链不完整，有线索价值但不作为最终结论 |
| `reference_only`：仅回溯资料 | 13 | 低置信、缺 log、集合入口或外部片段，只用于回溯来源 |
| `main_entry`：目录入口 | 12 | README、横向索引和专题入口，不计作真实 case |

## 分级规则

| 分层 | 判定口径 |
|---|---|
| A 类 | 高频业务、容易复现、能沉淀流程或配置规则，或者当前 `partial` 但明显有复用价值 |
| B 类 | 已经 `summarized` / `closed` / `reference`，能查到结论，当前不影响排障效率 |
| C 类 | 长尾单点、证据弱、附件型内容多，暂时只需要被搜索到 |
| 待合并 / 迁移 | 多个 case 实际属于同一专题，或业务域应迁到 IMS / Stability / 配置方法 |

## 证据缺口处理规则

`summarized_with_log_gap` 不代表不能用。它的使用方式是：

- 可复用已有判断方向和关键字段。
- 不把当前 case 当成最终 root cause 模板。
- 下次复现时按 case 内“下次复现补证清单”抓 AP / modem / 对比机 / 配置 / 产物证据。
- 补齐证据后再升级为 `summarized` 或 `closed`。

## 精修完成标准

一个 A 类 case 精修完成后，应满足：

1. `status` 从 `partial` / `open` / `summarized_from_search` 升到 `summarized`，除非仍缺关键证据。
2. `first_bad_point` 能指出第一处不满足，不写泛泛的“网络问题”。
3. 关键证据只保留能证明结论的 log / 字段 / 配置项。
4. 结论区明确区分事实、推断、待确认。
5. 有一个回填动作：流程、排障、配置、代码架构、工具方法至少更新一个。
