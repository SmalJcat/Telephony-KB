# ADR-003: 知识沉淀分层规则

## Status

Accepted

## Date

2026-05-18

## Context

通信问题定位资料会同时包含流程、平台实现、log字段、代码路径、真实case、工具命令和长期判断口径。如果都写进同一篇流程文档，文档会快速变长，并且后续很难判断新资料应该补在哪里。LTE注册流程整理后已经拆出平台代码架构和平台log速查，需要把这种分层方式固定为知识库规则。

## Decision

知识库按“入口索引、架构、流程、日志分析、案例、决策、工具、模板”分层维护。

| 内容类型 | 放置位置 | 判断标准 |
|---|---|---|
| 导航入口、知识地图、标签、术语 | `00_Index` | 帮读者决定先看哪里，不承载大段证据 |
| Android/平台代码架构、模块边界、平台差异 | `50_Platform-Code` | 回答“代码从哪里看、平台差异在哪里” |
| 正常流程、异常分叉、第一坏点判断口径 | `20_Service-Flows` | 回答“流程应该怎么走、坏在哪一层” |
| AP/modem log字段、trace模块、解码入口、时间线方法 | `70_Tools-Debug/Log-Analysis` | 回答“log怎么看、字段怎么对齐” |
| 真实问题、真实log证据链、结论和复盘 | `40_Case-Library` | 回答“这次问题发生了什么、下次如何复用” |
| 长期约定、分析边界、目录治理规则 | `90_Decisions` | 回答“以后统一按什么口径做” |
| adb/dumpsys/解析脚本/工具操作 | `70_Tools-Debug` | 回答“用什么命令或脚本检查” |
| 可复制模板 | `99_Templates` | 作为新文档起点 |

## Rules

- 主流程文档只写共性流程和判断口径，不贴大段原始log。
- 平台私有代码路径和状态同步链路写到 `50_Platform-Code`。
- 平台私有log字段、trace模块、工具入口写到 `70_Tools-Debug/Log-Analysis`。
- 真实log证据链写到 `40_Case-Library`，并在流程文档中只保留链接和简短结论。
- IMS/SIP/VoLTE内容不混入LTE注册流程；LTE注册只保留EPS注册和默认承载相关内容。
- 每个case必须有文件顶部frontmatter，至少包含 `domain`、`rat`、`platform`、`layer`、`symptom`、`cause`、`source_log`、`first_bad_point`、`confidence`、`status`。
- 新增重要判断口径时，如果会影响后续多个文档，先补 `90_Decisions` 或对应知识地图，再改具体文档。

## Consequences

- 流程文档会保持较短，但需要通过知识地图和case链接承接细节。
- case会成为长期复用的核心资产，写case时必须保证证据链完整。
- 后续可以用脚本自动检查frontmatter、断链、超大文档和主题混杂问题。
